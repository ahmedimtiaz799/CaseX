# CaseX ⚖️

### Talk to your legal documents. Get answers, not ambiguity.

---

## The Problem

Legal contracts are dense and time-consuming to read. A 40-page agreement can hide one clause that costs a business everything. Most people sign without fully understanding what they agreed to, not because they're careless, but because the right tooling has never been accessible to them.

CaseX is built to fix that.

---

## What It Does

CaseX is an AI-powered legal document analysis platform. Upload a contract, ask questions in plain English, and get precise answers pulled directly from your document.

It works for legal professionals who need speed without losing accuracy, and for anyone who wants to actually understand what they're signing before they sign it.

The system uses an Agentic RAG (Retrieval-Augmented Generation) pipeline. Every response is retrieved from your document, traced back to the exact page it came from, and generated by an AI persona configured to behave like a professional legal analyst.

---

## Core Features

**Secure Document Ingestion**
PDF uploads up to 10MB. The backend extracts, chunks, and embeds the document into a vector store tied to your session. No data crosses between sessions.

**Agentic Legal Persona**
Powered by Gemini 2.5 Flash, the AI agent only answers from your document. It maintains a professional tone and refuses to speculate outside the provided context.

**Source Attribution on Every Answer**
Every response includes the exact page numbers and document sections it referenced. The answers are verifiable, not just plausible.

**Session-Based Architecture**
Each upload creates a unique isolated session. Users can manage, switch between, and delete multiple review sessions from the sidebar, which is useful when working across several contracts at once.

**Contextual Suggestion Cards**
One-click prompts like "Summarize", "Identify Penalty Clauses", and "Check Termination Terms" activate once a document is uploaded, so users have an immediate starting point.

**Real-time Observability**
The full agent workflow is traced and monitored via Langfuse, giving visibility into retrieval quality, latency, and LLM behavior.

## System Architecture

CaseX is built around a LangGraph StateGraph with two core nodes:

```
User Query
    |
    v
[ Retrieve Node ]
  - Queries PGVector DB
  - Filters strictly by session_id
  - Returns top-k relevant chunks
    |
    v
[ Generate Node ]
  - Injects retrieved context + user query
  - Gemini 2.5 Flash generates response
  - System guardrails enforced via prompt
    |
    v
Structured Response + Page Citations
```

### Ingestion Pipeline

When a PDF is uploaded, it goes through three stages before any question can be answered:

1. **Extract** - `PyPDFLoader` pulls clean text from every page.
2. **Chunk** - `RecursiveCharacterTextSplitter` splits the text into 2,000-character chunks with a 400-character overlap to preserve context at boundaries.
3. **Embed & Store** - `all-MiniLM-L6-v2` (HuggingFace) generates vector embeddings stored in PostgreSQL via the `pgvector` extension. Each chunk is tagged with a `session_id` for strict data isolation.

## Getting Started

### Prerequisites

- Python 3.11+
- Node.js 18+
- PostgreSQL with the `pgvector` extension enabled
- A Supabase project (for Auth)
- A Google AI Studio API key (Gemini)
- A Langfuse account (optional, for observability)

### 1. Clone the Repository

```bash
git clone https://github.com/your-username/casex.git
cd casex
```

### 2. Backend Setup

```bash
cd backend
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 3. Frontend Setup

```bash
cd frontend
npm install
```

### 4. Configure Environment Variables

Create a `.env` file in the `backend/` directory (see the section below).

### 5. Run the Application

```bash
# Terminal 1 - Backend
cd backend
uvicorn main:app --reload --port 8000

# Terminal 2 - Frontend
cd frontend
npm run dev
```

The app runs at `http://localhost:5173`.

## Environment Variables

Create a `.env` file in `backend/` with the following:

```env
# Google Gemini
GOOGLE_API_KEY=your_google_api_key

# Supabase
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key
SUPABASE_JWT_SECRET=your_jwt_secret

# PostgreSQL + pgvector

LANGFUSE_PUBLIC_KEY=your_langfuse_public_key
LANGFUSE_SECRET_KEY=your_langfuse_secret_key

```

## Why This Architecture?

Most AI document tools dump the entire document into a prompt and rely on the model to figure out the rest. That approach breaks down with long contracts and gives you no way to verify where an answer came from.

CaseX uses session-isolated vector retrieval so only your document's chunks are ever in scope. The LangGraph StateGraph keeps retrieval and generation as separate, traceable steps. The system prompt guardrails mean the model cannot fabricate an answer, it either finds it in your document or tells you it is not there.
